3. Search – Keyword, Metadata, Full Text & Advanced
Keyword Search

Users can run keyword searches from the Documents tab (and other dynamic object tabs). These are standard full-text/metadata queries against the SQL full-text index. Supports Boolean operators and wildcards. 
help.relativity.com
+1

Example: “implant failure” AND (device* OR “medical device”).

dtSearch Index

In addition to the keyword index, Relativity supports a dtSearch index which provides advanced search features: proximity, fuzziness, stemming, and wildcarding. Admins can create dtSearch indexes for specific document sets and assign security. 
help.relativity.com

Analytics / Conceptual Search

Beyond keyword/dtSearch, Relativity includes Analytics search functionality (conceptual searching, clustering) that finds documents based on conceptual similarity rather than explicit terms. 
help.relativity.com

This is valuable when you want to find documents about an idea or theme (e.g., “adverse event communication”) rather than specific words.

Regular expression (RegEx) searches may also be supported (via dtSearch) for pattern matching (e.g., serial numbers, device IDs).

+1

Saved Searches

You can create and save complex search queries (Saved Searches) which can include conditions on any field (metadata), index search (keyword/dtSearch/Analytics), folder scope, etc. These saved searches can be reused, shared, organized in folders, and can be set to “requires manual rerun” to control performance. 
help.relativity.com
+1

In the Saved Search definition, you can specify: Name, Owner (public/private), Scope (entire workspace or selected folders) and set logic groups/conditions. 
help.relativity.com

Search Conditions & Operators

When building a saved search (or advanced search) you define conditions: Field + Operator + Value. Operators vary by field type (text, number, date, choice). For example: “Extracted Text Contains (‘device malfunction’)”, or “Date > 2020-01-01”. 
help.relativity.com

Avoid over‐complex nested saved searches (they may timeout or cause performance issues). 
help.relativity.com

Scope & Result Sets

Searches execute dynamically: you define the criteria, save the search, then when you run it the system returns the live set of documents matching those criteria (not a static snapshot).

You can configure whether related items (e.g., family documents) are included in the result set via the “Includes Family” or related-items setting in search.

4. Integration of Review + Search + Coding

Bringing it all together, the workflow typically proceeds as:

Configure document list/view: decide which columns (fields) appear, what filters are default.

Use search (keyword, dtSearch, Analytics) or saved search to isolate a relevant subset of documents (e.g., all device implant cases from 2018-2023).

Apply filters further on field values (e.g., DeviceStatus, Custodian, ReviewStatus).

Launch Review Interface on the selected documents (either via batch, queue, or list).

Reviewer opens document in Viewer, uses intra-doc search to locate key terms, then applies coding via layout (e.g., marking “AdverseEvent = Yes”, “Severity = High”).

Use Related Items card to capture attachments/family docs and apply consistent coding across the set.

Save coding decisions, then move to next document (or Save & Next).

Throughout, admins/managers can run dashboards or saved searches that monitor progress (e.g., number of “High Severity” flagged docs, percentage reviewed).

Export or produce documents based on coded fields (e.g., all docs tagged “AdverseEvent = Yes” for further analysis or legal production).

5. Tips & Considerations

Indexing: Ensure extracted text indexing (keyword, dtSearch) is set up and kept current; full-text search performance depends on good indexing.

View design: For large datasets (e.g., medical records + device logs + email communications) tailor views so only necessary columns display to reduce clutter.

Coding consistency: Use multi-choice fields, default values, and “Copy from Previous” where applicable to maintain consistency among reviewers.

Performance: Avoid overly complex nested saved searches or extremely broad dtSearches (they may slow environments). Use logic groups efficiently. 
help.relativity.com

Security & permissions: Properly configure permissions so only appropriate users can view/edit coding fields, view privileged docs, etc.

Family groups: Make sure family relationships (e.g., email + attachment + excel log) are identified so reviewers see full context and the “Includes Family” option is used when searching.

Audit/tracking: Review history is captured (who coded, when, what changes) so you have accountability and defensibility.

Training & review template setup: Especially for a specialized domain like medical records, you’ll want to define coding schemas, layout templates, reviewer instructions, highlight sets (e.g., terms like “implant”, “failure”, “recall”) so the system supports domain filtering.

Analytics + concept search: Since medical/device data may use varied terminology, conceptual/cluster search (via Analytics) can surface hidden connections (e.g., similar cases where “device extraction” is implied but not spelled out).
