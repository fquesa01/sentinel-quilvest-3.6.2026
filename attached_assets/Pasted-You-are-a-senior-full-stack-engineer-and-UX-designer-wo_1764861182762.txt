You are a senior full-stack engineer and UX designer working on a web app called **Sentinel** (an investigations, compliance, and e-discovery platform).

Your job:
Upgrade the existing AI assistant **“Ava – AI Compliance Assistant”** so that users can **speak or type natural language commands to navigate Sentinel and perform tasks**, while still being able to ask compliance / legal questions in the same interface.

IMPORTANT: EXISTING AVA UI

The current Ava experience already exists and must be extended, not replaced:

- There is an **Ava button in the bottom-left** of the sidebar (purple button with Ava label).
- On case pages (e.g., Findings tab), there is an **“Ask Ava” button** in the main content area.
- Clicking either opens a **right-side drawer/panel** labeled “Ava – AI Compliance Assistant.”
- The panel currently shows:
  - Ava header with icon + name
  - “New Chat” button
  - Welcome text and example prompts
  - A **single text input at the bottom**: “Ask Ava anything…” with a send icon.

You must keep this **right-side drawer layout** and integrate voice + command functionality directly into this existing Ava panel and input.

================================
1) HIGH-LEVEL GOAL
================================

Ava should work like a smart secretary for the user:

1) Users can **navigate the app** by speaking or typing commands.
2) Users can **execute tasks** like scheduling interviews.
3) Users can still **ask legal/compliance questions** and get normal chat responses.
4) Ava decides if an input is:
   - A **command** (navigate/perform an action), or
   - A **knowledge Q&A** (answer with text only),
   and routes appropriately.

App context (simplified):
- My Queue (home)
- Investigations:
  - Cases
  - Email & Document Review
  - Chat Conversations
  - Interviews
  - Document Sets
- Intake & Monitoring:
  - Communications

================================
5) COMMAND vs. Q&A ROUTING
================================

When the user submits a message (typed or spoken), Ava must decide:

- Is this a **command** (navigate or perform a task)?
- Or a **knowledge/compliance question**?

Implementation:

1) Front-end sends the message + context to a new back-end endpoint:
   - `POST /api/ava/interpret`
   - Input:
     - `message: string`
     - `context: { userId, currentRoute, currentCaseId?, currentCaseName?, currentTab?, timezone, ... }`

2) Back-end uses an LLM to:
   - Classify the message as:
     - `"mode": "command"` or `"mode": "qa"`.
   - If `"command"`, return:
     ```json
     {
       "mode": "command",
       "intent": "navigate_to_case_email_filter",
       "parameters": {
         "caseName": "USA Water Polo",
         "emailFilter": {
           "participants": ["Frank", "Chris Ramsey"]
         }
       },
       "assistantMessage": "Opening the USA Water Polo case and filtering to emails between Frank and Chris Ramsey."
     }
     ```
   - If `"qa"`, return:
     ```json
     {
       "mode": "qa",
       "assistantMessage": "…Ava’s compliance/legal answer here…"
     }
     ```

3) On the front-end:
   - If mode = `qa`:
     - Simply render `assistantMessage` as the chat response.
   - If mode = `command`:
     - Run the **command router** (see next section) and ALSO render `assistantMessage` in the chat as confirmation.

================================
6) COMMAND ROUTER – NAVIGATION & ACTIONS
================================

Implement a front-end **commandRouter** that takes `{ intent, parameters }` and performs actions.

Supported intents (minimum):

- Navigation:
  - `navigate_to_case`
  - `navigate_to_case_document_review`
  - `navigate_to_case_email_filter`
  - `navigate_to_my_queue`
  - `navigate_to_interviews`
  - `navigate_to_chat_conversations`
  - `navigate_to_document_sets`
  - `navigate_to_findings`

- Actions:
  - `schedule_interview`

Examples of commands the interpreter should understand:

1) Navigation to case & filtered emails
   - User: “Take me to the USA Water Polo case, I want to review Frank’s emails with Chris Ramsey.”
   - Interpret:
     - `intent: "navigate_to_case_email_filter"`
     - `parameters: { caseName: "USA Water Polo", emailFilter: { participants: ["Frank", "Chris Ramsey"] } }`
   - Router:
     - Call `/api/cases/search?name=USA%20Water%20Polo`.
     - Resolve to `caseId`.
     - Navigate to `/cases/:caseId/email-document-review` with appropriate query params or filter state.
   - Ava chat: “Done. I’ve opened USA Water Polo and filtered emails between Frank and Chris Ramsey.”

2) Jump to Interviews from within a case
   - User (on a case): “I need to schedule an interview.”
   - Context includes `currentCaseId`.
   - Interpret:
     - `intent: "navigate_to_interviews"`
     - `parameters: { caseId: "<currentCaseId>" }`
   - Router:
     - Navigate to `/cases/:caseId/interviews`.
   - Ava: “Opening the Interviews tab for [Case Name].”

3) Go to My Queue
   - User: “Take me to my queue.”
   - Interpret:
     - `intent: "navigate_to_my_queue"`
   - Router:
     - Navigate to `/my-queue`.
   - Ava: “Taking you to your queue.”
